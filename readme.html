Step 1 — Configure the Waveshare RS485-to-ETH Gateway

The Waveshare 8-CH RS485 TO POE ETH (B) has 8 independent RS485 channels. Connect the Eastron meter to channel 1 (or any channel — note which one you use).

1a
Find the gateway IP. Look in your router's DHCP client list, or scan your subnet:
nmap -sn 192.168.1.0/24  (replace with your subnet)
1b
Open the web interface. Navigate to http://<gateway-ip>. Default: no username or password.
1c
Configure the channel. Find settings for the port connected to the Eastron meter (e.g. "UART1"):
Work ModeModbus TCP to RTU
Local Port502
Baud Rate9600
Data Bits8
Stop Bits1
ParityNone
Save and reboot the gateway if prompted.
1d
Verify port 502 is open:
telnet <gateway-ip> 502
A blank screen means success (press Ctrl+] then type quit). "Connection refused" means the Work Mode is wrong.

Step 2 — Wire the Eastron Meter

RS485 communication (to Waveshare)

Eastron terminalWaveshare terminal
A+Channel A
B−Channel B

Power and measurement (1-phase 2-wire — page 11 of the Eastron manual)

WireEastron terminalPosition
Mains phase (L)Terminal 1Top-left
Mains neutral (N)Terminal 4Top-right
Load phase outTerminal 5Bottom-left
Load neutral outTerminal 8Bottom-right
Terminals 1+4 power the meter and measure voltage.
Current is measured by current flowing through the meter: in at 1+4, out at 5+8.
Wire your load between terminals 5+8 and the rest of your circuit.
If nothing is connected to 5+8, current and power will read 0 — that is normal.

After powering on, the Eastron screen shows 00000.00 (energy counter at zero). If the screen is blank, check mains wiring and terminal screw tightness.

Step 3 — Install Mosquitto MQTT Broker in Home Assistant

The primary data path (Modbus YAML, Step 5) does not need MQTT. Only do this step if you also want the MQTT bridge (Step 7).
3a
In HA: Settings → Add-ons → Add-on Store
Search Mosquitto broker → Install
3b
Click Start → toggle Start on boot and Watchdog on → Start
3c
Create an MQTT user: Settings → People → Users → Add User
Example: username mqtt, password yourpassword. Note these for Step 7.
3d
If HA does not auto-discover MQTT: Settings → Devices & Services → Add Integration → MQTT
Host: localhost   Port: 1883   Username/Password: as created above

Step 4 — Set Up SSH Access to Home Assistant

You need SSH to copy modbus_eastron.yaml onto the HA filesystem and to edit configuration.yaml. This only needs to be done once.

4a
Install the SSH addon:
Settings → Add-ons → Add-on Store → search Advanced SSH & Web Terminal → Install
4b
Configure key-based authentication (recommended).
On your machine, generate a key if needed: 
ssh-keygen -t ed25519
Copy your public key: 
cat ~/.ssh/id_ed25519.pub
In HA: Settings → Add-ons → Advanced SSH → Configuration tab, add: 
authorized_keys:
  - "ssh-ed25519 AAAA... you@yourmachine"
Click Save.

Alternatively, set a password in the same tab: password: "yourpassword"
4c
Click Start → enable Start on boot and Watchdog
4d
Connect: 
ssh hassio@<HA_IP>
Username is always hassio — not your HA login name, not "root". Port is 22 (default).

Step 5 — Deploy Modbus Sensor Config to Home Assistant

This creates 34 sensor entities in HA that poll the Eastron meter every 30 seconds (60 s for energy counters).

5a
Edit modbus_eastron.yaml — change the host to your Waveshare channel IP: 
host: 192.168.0.22    # ← replace with your gateway IP
If you changed the meter's Modbus address (default is 1), also update: 
slave: 1              # ← must match the meter's address
5b
Copy the file to Home Assistant: 
scp modbus_eastron.yaml hassio@<HA_IP>:/config/modbus_eastron.yaml
5c
Add the include line to configuration.yaml: 
ssh hassio@<HA_IP>
nano /config/configuration.yaml
Add this line at the root level: 
modbus: !include modbus_eastron.yaml
Save with Ctrl+O → Enter, exit with Ctrl+X.
modbus_eastron.yaml must not start with modbus:.
It must start with: - name: "eastron_sdm72dm"
The modbus: key comes from configuration.yaml via the !include directive.
5d
Restart Home Assistant: 
ha core restart
Or: Settings → System → Restart → Restart Home Assistant.
HA will be unavailable for ~30–60 seconds.
5e
Verify sensors are live:
Settings → Devices & Services → Entities → search sdm72d
You should see 34 entities. Within 30 seconds they will all show values.

Step 6 — Deploy the Lovelace Dashboard

Run deploy_dashboard.sh from any machine on the same network as HA. Requirements: bash, curl, python3 (standard on any Linux/Mac). dashboard.json must be in the same directory.

6a
Edit deploy_dashboard.sh — set the three variables at the top: 
HA_HOST="192.168.0.4"   # your HA IP address
HA_USER="admin"         # your HA username
HA_PASS="yourpassword"  # your HA password
6b
Run the script: 
bash deploy_dashboard.sh
Successful output ends with: 
Dashboard deployed successfully.
Open: http://<HA_IP>:8123/lovelace
To customise the dashboard layout, edit dashboard.json and re-run the script.
The script uses python3 internally only for the WebSocket transport (HA does not expose the Lovelace save API over REST). All configuration and logic is in bash.
6c
Open the dashboard: http://<HA_IP>:8123/lovelace
TabContents
OverviewGauge dials (voltage, frequency, power, current, PF) + totals list + averages
Per PhasePer-phase gauge dials + glance cards for L-L voltages, power, reactive power, PF
Energy HistoryImport/export statistics + 24h voltage and current graphs
StatisticsDaily/monthly bar charts for energy and power

Step 7 — Install the MQTT Bridge Service (Optional)

The MQTT bridge is a secondary/backup data path. The Modbus sensors from Step 5 are the primary method. Only set this up if you want redundancy or cannot use the Modbus YAML approach.

The bridge polls the meter via Modbus TCP and publishes values to Mosquitto on HA. HA creates MQTT sensors automatically via auto-discovery.

7a
Edit eastron_mqtt_bridge.py — set at the top: 
MODBUS_HOST = "192.168.0.22"   # Waveshare channel IP
MQTT_HOST   = "192.168.0.4"    # HA IP
MQTT_USER   = "mqtt"           # MQTT username (from Step 3)
MQTT_PASS   = "yourpassword"   # MQTT password
7b
Edit eastron_bridge.service — update the path: 
ExecStart=/usr/bin/python3 /path/to/eastron_mqtt_bridge.py
7c
Copy both files to the machine that will run the bridge (your Linux VM, Raspberry Pi, etc.)
7d
Install the systemd user service: 
mkdir -p ~/.config/systemd/user/
cp eastron_bridge.service ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable eastron_bridge
systemctl --user start eastron_bridge
7e
Enable auto-start on boot (without needing to log in): 
sudo loginctl enable-linger $USER
7f
Verify it is running: 
systemctl --user status eastron_bridge
journalctl --user -u eastron_bridge -f    # live log, Ctrl+C to exit

HA does not backfill statistics. Data accumulates from the moment the sensors were first created. There is no history before that point.

To add a statistics-graph card manually (Add Card → Manual):

type: statistics-graph
title: Energy Import / Export (Monthly)
period: month
stat_types:
  - sum
entities:
  - entity: sensor.sdm72d_import_energy
    name: Import
  - entity: sensor.sdm72d_export_energy
    name: Export

Adding More Eastron Meters

Scenario A — Same RS485 bus, different slave addresses Daisy-chain

All meters share one Waveshare channel. Each meter needs a unique Modbus slave address set on the meter itself (factory default is 1). Change addresses via the front panel buttons before wiring meters together.

# Add to modbus_eastron.yaml — meter 2, slave address 2
- name: "eastron_meter2"
  type: tcp
  host: 192.168.0.22        # same Waveshare channel
  port: 502
  sensors:
    - name: "M2 L1 Voltage"
      unique_id: m2_sdm72d_l1_voltage
      slave: 2              # ← meter 2 address
      address: 0
      input_type: input
      data_type: float32
      unit_of_measurement: "V"
      device_class: voltage
      state_class: measurement
      scan_interval: 30
    # ... repeat all 34 registers with slave: 2

Entity IDs: sensor.m2_l1_voltage, sensor.m2_l2_voltage, …

Scenario B — Different Waveshare channels, each with slave address 1 Separate channels

Each meter connects to its own Waveshare channel (separate A/B pair). All meters can keep slave address 1.

# Add to modbus_eastron.yaml — meter on CH2
- name: "eastron_ch2"
  type: tcp
  host: 192.168.0.31        # Waveshare CH2 IP
  port: 502
  sensors:
    - name: "CH2 L1 Voltage"
      unique_id: ch2_sdm72d_l1_voltage
      slave: 1
      address: 0
      input_type: input
      data_type: float32
      unit_of_measurement: "V"
      device_class: voltage
      state_class: measurement
      scan_interval: 30
    # ... all 34 registers

Entity IDs: sensor.ch2_l1_voltage, sensor.ch2_l2_voltage, …

Rules for multiple meters:
• Every unique_id must be globally unique — use a clear prefix per meter (m1_, m2_, kitchen_, garage_, etc.)
• After editing the YAML, restart HA: ha core restart or Settings → System → Restart

Troubleshooting

Meter screen is completely blank

  • Mains circuit feeding terminals 1+4 is not live (check breaker)
  • Terminal screws are loose — re-insert wire and tighten firmly

Sensors show "unavailable" in HA

ping <gateway-ip>                         # is Waveshare reachable?
telnet <gateway-ip> 502                    # is port 502 open?
ssh hassio@<HA_IP>
grep -i modbus /config/home-assistant.log  # check for errors

Sensors disappeared after HA restart

ssh hassio@<HA_IP>
cat /config/configuration.yaml    # must contain: modbus: !include modbus_eastron.yaml
head -3 /config/modbus_eastron.yaml  # must start with: - name:  (NOT with modbus:)

deploy_dashboard.py fails with "Auth failed"

  • Double-check HA_HOST, HA_USER, HA_PASS at the top of the script
  • Make sure HA is reachable: ping <HA_IP>
  • Try opening http://<HA_IP>:8123 in a browser first

MQTT bridge stopped

systemctl --user restart eastron_bridge
journalctl --user -u eastron_bridge -f
