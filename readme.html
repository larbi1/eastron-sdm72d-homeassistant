The Waveshare gateway bridges RS485 serial (Eastron) to Modbus TCP (Ethernet).

Waveshare CH1 settings: Modbus TCP→RTU, port 502, baud 9600, 8N1, no parity, no password.

Eastron settings: Modbus slave address 1, baud 9600, 8N1. All registers use function code 04 (Read Input Registers), values are 32-bit floats.

How Home Assistant Receives Data

Two parallel methods are active simultaneously:

Method 1 — Modbus YAML Primary

HA polls the Waveshare directly via Modbus TCP, configured in /config/modbus_eastron.yaml.

Included in /config/configuration.yaml via: modbus: !include modbus_eastron.yaml

Poll interval: every 30 seconds (60 s for energy counters).

Entity IDs: sensor.sdm72d_*   e.g. sensor.sdm72d_l1_voltage

The Lovelace dashboard uses these entity IDs.

Method 2 — MQTT Bridge Backup

A Python script (eastron_mqtt_bridge.py) runs as a systemd service on 192.168.0.17.

It polls Modbus TCP → publishes values to the Mosquitto MQTT broker on HA → HA auto-discovers sensors.

Entity IDs: sensor.eastron_sdm72d_m_sdm72d_*

systemctl --user status eastron_bridge    # check status
systemctl --user restart eastron_bridge   # restart
journalctl --user -u eastron_bridge -f    # live logs

Multiple Eastron Meters

Two scenarios depending on how you connect the additional meters.

Scenario A — Same RS485 bus, different slave addresses Daisy-chain

All meters share one Waveshare channel. Each meter must have a unique Modbus slave address set on the meter itself (factory default is 1 for all).

Change the address on each meter via its front panel buttons before wiring them together on the same A/B bus.

In /config/modbus_eastron.yaml, add a separate hub block per meter (or reuse the same hub with different slave: numbers):

# Meter 1 — slave address 1 on Waveshare CH1
- name: "eastron_meter1"
  type: tcp
  host: 192.168.0.22
  port: 502
  sensors:
    - name: "M1 L1 Voltage"
      unique_id: m1_sdm72d_l1_voltage
      slave: 1
      address: 0
      input_type: input
      data_type: float32
      unit_of_measurement: "V"
      device_class: voltage
      state_class: measurement
      scan_interval: 30
    # ... repeat all 34 registers with slave: 1

# Meter 2 — slave address 2, same Waveshare channel
- name: "eastron_meter2"
  type: tcp
  host: 192.168.0.22
  port: 502
  sensors:
    - name: "M2 L1 Voltage"
      unique_id: m2_sdm72d_l1_voltage
      slave: 2
      address: 0
      input_type: input
      data_type: float32
      unit_of_measurement: "V"
      device_class: voltage
      state_class: measurement
      scan_interval: 30
    # ... repeat all 34 registers with slave: 2

Entity IDs: sensor.m1_l1_voltage, sensor.m2_l1_voltage, …

Each meter connects to its own Waveshare channel (separate A/B pair). All meters can keep slave address 1 (factory default).

Important rules for multiple meters:
• Every unique_id must be globally unique — use a clear prefix per meter (m1_, m2_, kitchen_, garage_, etc.)
• After editing the YAML, SSH into HA and run ha core restart, or use HA UI: Settings → System → Restart.
• Each meter adds 34 sensors. With 8 meters you get 272 sensors — increase scan_interval to reduce Modbus bus load.

Troubleshooting

Meter screen is completely blank

The meter has no power. Check:

  • Mains circuit feeding terminals 1 and 4 is live (breaker on)
  • Wires are fully inserted and terminal screws are tightened

Sensors show "unavailable" in HA

ping 192.168.0.22                     # is Waveshare reachable?
ssh hassio@192.168.0.4
grep -i modbus /config/home-assistant.log  # check for errors

Modbus sensors disappeared after HA restart

ssh hassio@192.168.0.4
cat /config/configuration.yaml        # must contain: modbus: !include modbus_eastron.yaml
head -3 /config/modbus_eastron.yaml   # must start with: - name:  (NOT with modbus:)

MQTT bridge stopped

systemctl --user restart eastron_bridge
journalctl --user -u eastron_bridge -f

Make MQTT bridge start on boot without login

sudo loginctl enable-linger akaw
